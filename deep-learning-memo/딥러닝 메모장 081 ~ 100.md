Cross-Entropy Loss: 정답을 틀렸을 때 모델에게 매기는 벌금(Loss)
하지만, 그냥 벌금을 매기는 게 아니다. "모델이 자신만만하게(높은 확률로) 정답이라고 확신했는데 틀렸을 때"
벌금을 거하게 때린다. 반대로 낮은 확률(예: 0.6 정도)로 틀리면 벌금을 적게 낸다.

로그(log) 함수가 들어가는 이유.
-log(x) 그래프를 떠올리면 1(100% 확신)에 가까울수록 값이 0에 수렴하고 0(0% 확신)에 가까울수록 값이 무한대로 향한다.

케이스 1(착한 모범생): 99% 확률로 신선한 사과라고 추측
	결과: 정답을 맞혔고, 확신도 높았다.
	벌금(Loss): -log(0.99) ~= 0.01 (벌금이 거의 없음)

케이스 2(공부 덜 한 녀석): 신선한 사과일 확률 40%, 오렌지일 확률 60%로 추측
	결과: 정답(사과)을 틀렸고, 사과에 대한 확신이 40%에 그쳤다.
	벌금(Loss): -log(0.4) ~= 0.91 (정답 확률이 낮아진 만큼 벌금이 조금 올라감)
	
케이스 3(자신만만한 바보): 99.9% 확률로 상한 오렌지. 신선한 사과일 확률은 0에 가깝다고 추측
	결과: 정답은 사과이고, 거의 0%로 예측했다.
	벌금(Loss): -log(0.001) ~= 6.9 (벌금을 오지게 때려버림)

모델이 예측한 정답 클래스의 확률값에 -log(x)를 취해 손실을 계산함. 오답을 높은 확률로 확신할수록 Loss가 기하급수적으로 늘어남. 이렇게 함으로써 단순히 틀리는 것을 넘어 '오만하게 틀리는 것'을 강력히 억제하고 정답 확률을 100% 가깝게 끌어올리는 게 목표임.

모델이 예측한 결과가 [10.5, 9.8, 0.1, -2.3, -5.0, 1.2]일 때, 
시그모이드를 태웠을 때 앞 두 개는 확률이 99.9%, 99.8% 이런 식으로 튀어나올 것이다.
즉, 모든 클래스의 확률을 더했을 때 100%(1.0)이 되지 않는 문제가 생긴다.
얼마나 높은 확신도를 가지고 맞혔는지 알 수가 없다.

따라서 시그모이드는 멀티 라벨 분류나 이진 분류에만 쓴다.

파이썬 iloc, loc의 차이
가장 핵심정인 차이는 데이터를 지정하는 방식(이름 vs 위치)과 슬라이싱 범위다.

loc(location)은 이름(label) 기반 접근이고, iloc(Integer Location)은 정수 인덱스(position) 기반 접근이다.

[코드 예시]
import pandas as pd

data = {'수학': [90, 80, 85], '영어': [100, 90, 95]}
df = pd.DataFrame(data, index=['a', 'b', 'c'])

print(df.loc['a', '수학']) # 90
print(df.iloc[0, 0]) # 90

print(df.loc['a':'b']) # 이름이 'a'인 행부터 'b'인 행까지 모두 포함하여 출력한다.
print(df.iloc[0:2]) # 0번째 행부터 2 직전인 1번째 행까지만 포함하여 출력한다.

# 수학 점수가 85점 이상인 행의 '영어' 컬럼만 조회

위치(row_num) vs 고유 인덱스(label)

위치(row_num): 줄을 재정렬할 때마다 계속 바뀌는 현재 순서 (iloc이 바라보는 기준)
고유 인덱스(label): 데이터가 정렬되거나 섞여도 변하지 않는 고유 ID (loc이 바라보는 기준)
